# 图片标注工具 PRD（v0.1 草案）

## 1. 文档信息
- 产品名称（暂定）：简标 PicMark
- 平台：Windows 10 / 11（桌面应用）
- 文档版本：v0.1
- 状态：草案，待评审

## 2. 背景与目标

用户经常需要在截图或照片上做简单标注（比如给同事/家人解释一个表格画个框、给客服反馈问题截图加文字说明、给报表截图圈重点），但现有工具要么功能太复杂（PS、Photoshop Elements），要么是网页工具不支持离线大图、容易压缩失真，要么操作路径太长、界面太丑，不适合非专业修图用户日常顺手使用。

**目标：** 做一个 Windows 上的轻量标注工具，核心卖点两个：
1. **画质无损** —— 标注后导出的图片，原图部分像素不被压缩/缩放/重新编码劣化，标注图层精确叠加在原始分辨率坐标上。
2. **简单好用、好看** —— 普通中年非专业用户也能在 1 分钟内学会"打开图片 → 画个圈/框/加字 → 保存"全流程，不需要看说明书；界面参考主流大厂产品的现代化视觉风格，而非堆砌按钮的工具箱观感。

## 3. 目标用户与典型场景

| 用户 | 场景 |
|---|---|
| 普通办公用户 | 给报表截图画框强调数据，发邮件 |
| 客服/技术支持人员 | 在用户截图上标注问题区域 |
| 远程指导家人朋友 | 截图后圈出按钮位置，配文字说明，发微信 |
| 普通用户日常标注 | 给照片做简单标记（如圈出体检报告异常值）后打印或发送 |

共同特点：**不追求专业修图能力，追求"打开-标-保存"三步走、不出错、不毁原图、界面顺眼**。

## 4. 核心设计原则

1. **原图不可被破坏**：任何操作默认不覆盖原文件，另存为新文件；标注前后原图字节不变。
2. **标注即矢量、导出才合成**：标注（圆、框、文字等）在编辑过程中始终是可移动、可删除、可改色的独立图层对象，不在编辑阶段烧录进位图，避免反复操作导致的多次重新编码损失。
3. **坐标对齐原始分辨率**：无论屏幕显示时图片被缩放多大比例查看，标注的位置/大小都按原始像素坐标记录和导出，保证导出图与原图分辨率一致、标注位置精确。
4. **导出无损优先**：
   - PNG/BMP/原本无损格式 → 合成后依然无损保存。
   - JPEG 原图 → 提供"保持原格式（最高质量重新编码）"与"另存为 PNG（完全无损）"两个选项，并在 UI 上用大白话提示用户哪个更不损画质。
5. **三步交互**：打开 → 选工具标注 → 保存。每一步都不超过 2 次点击，没有弹窗式的复杂参数表单。

## 6. 关键用户流程

```
打开图片
  → 鼠标拖拽图片到窗口 / Ctrl+V 粘贴 / 双击文件
选择工具
  → 顶部工具栏点选"圆/框/文字"图标（大图标 + 中文文字标签，不只是图标）
绘制标注
  → 鼠标拖拽出图形 / 点击输入文字，颜色/粗细已有合理默认值，不强制设置
（可选）调整
  → 点选已画图形可拖动、改色、删除
保存
  → 点击"保存"大按钮 → 默认另存为新文件，弹出极简提示"已保存为 XXX_标注.jpg，原图未改动"
```

全程无需阅读说明书，所有按钮使用图标 + 中文文字标签的组合（而非纯图标猜测，也不是纯文字的工具箱式按钮），主按钮高度、点击区域参照主流桌面应用的常规尺寸（约 44-56px 高），保证好点击但不刻意做成"放大版"。

## 7. 易用性与视觉设计要求

1. **大而少**：主界面按钮数量控制在合理范围内，按功能分组（绘图工具 / 撤销保存 / 颜色粗细字号），用分隔线区分，避免功能堆砌成一排。
2. **图标 + 中文文字双重提示**，图标用简洁矢量线条风格（不依赖系统图标字体，保证 Win7 也能正常显示），不单独依赖图标含义。
3. **现代化视觉风格**：参考主流大厂产品（如 Notion、Linear 等）的扁平化设计语言——圆角按钮、克制的配色（中性灰底 + 单一强调色）、清晰的悬停/选中状态反馈，而非传统 Windows 工具箱的灰色立体按钮。
4. **高对比度配色**，默认浅色主题，文字对比度达到 WCAG AA 以上。
5. **容错**：任何操作均可撤销；保存默认不覆盖原图，杜绝"误操作毁掉原图"的恐惧感。
6. **无需联网、无需注册登录**，打开即用。
7. **操作反馈明显**：保存成功/失败用弹窗 + 简短文案提示（如"已保存好啦"），而非技术性英文报错。
8. **快捷键对齐主流图像软件习惯**（如 Photoshop）：Ctrl+Z 撤销、Ctrl+Shift+Z/Ctrl+Y 重做、Ctrl+S 保存、Ctrl+O 打开、Ctrl+加减号缩放、Ctrl+0 适应窗口、Ctrl+1 实际大小、Delete 删除选中、Ctrl+D 取消选中，降低有其他软件使用经验的用户的学习成本。
9. **支持鼠标为主，兼容触屏**（如果是触屏笔记本/一体机，手指点选也要好用，按钮间距足够大避免误触相邻按钮）。

## 9. 技术方案（已定稿，因 Win7 兼容性要求收窄选型）

由于用户测试环境包含 Windows 7，以下方案被排除：
- **WinUI 3 / 现代 UWP**：仅支持 Windows 10 1809+，不支持 Win7。
- **.NET 6/7/8（含 .NET 6+ 上的 WPF）**：.NET 6 起官方运行时已不支持 Windows 7。
- Electron 理论上能跑在 Win7，但体积大、与"轻量、适老化原生体验"的目标冲突，且 Win7 上 Chromium 内核较旧、兼容风险更高。

**最终选型：WPF + .NET Framework 4.7.2**
- .NET Framework 4.7.2 最低要求 Windows 7 SP1，同时也是 Win10/Win11 上长期支持、开箱即用的版本（系统自带或极易安装，无需用户额外装 .NET 运行时）。
- 沿用经典（非 SDK-style）WPF 项目格式，便于在 Visual Studio 2019/2022 中直接打开编译，无需 .NET SDK。
- 原生 Win32/WPF 渲染，文件关联、右键菜单"使用 PicMark 打开"集成自然，安装包可控制在较小体积。

- **图像处理**：
  - 底图始终以原始分辨率位图（或直接引用原文件解码结果）保存在内存中，不做有损中间转换。
  - 标注对象（圆/框/文字/箭头）以矢量对象列表形式维护（位置、大小、颜色、线宽均按原图像素坐标存储）。
- **图像处理**：
  - 底图始终以原始分辨率位图（或直接引用原文件解码结果）保存在内存中，不做有损中间转换。
  - 标注对象（圆/框/文字/箭头）以矢量对象列表形式维护（位置、大小、颜色、线宽均按原图像素坐标存储）。
  - 屏幕显示时对底图+矢量层做统一缩放渲染（仅影响显示，不影响底层数据）。
  - 导出时按原始分辨率合成（底图原始像素 + 矢量层栅格化叠加），一次性编码输出，避免多次重复压缩。
- **文件处理**：默认另存为新文件，文件名自动加后缀防覆盖；提供"覆盖原图"显式选项并二次确认。

## 10. 成功指标（验收标准）

1. 找 5 位非专业修图、日常使用电脑办公的中年用户做可用性测试，**90% 能在无人指导下 3 分钟内完成"打开图片→画圈→加字→保存"全流程**。
2. 抽样导出图片与原图做像素级比对（PNG 场景）：标注以外区域像素完全一致；JPEG 场景：肉眼及客观指标（如 PSNR）无可感知劣化。
3. 4000 万像素以内照片，标注操作（拖拽绘制、移动图形）流畅无明显卡顿。
4. 用户测试中"误覆盖原图""找不到保存按钮""不知道怎么撤销"等问题反馈为 0 或可忽略。

## 12. 看图浏览器模块（新功能规划，参考 2345看图王）

> 状态：大纲已讨论定稿（2026-06-18），尚未开始实现。本节描述的是在现有标注/裁切编辑器之外新增的一个"看图浏览"入口与窗口，与第 1-11 节描述的编辑器是同一个 exe 内的两个模式，不是独立产品。

### 12.1 背景与目标

现有 PicMark 是"打开单张图 → 标注/裁切 → 保存"的编辑器，但用户日常打开图片的第一动作往往是"双击文件先看一眼"，而不是直接进入编辑状态——尤其是批量裁切功能跑完之后，用户需要快速翻看"已裁剪"文件夹里的结果，现有编辑器没有这个浏览能力，要么挨个双击打开很慢，要么得跳到资源管理器里看缩略图。

**目标**：新增一个轻量的图片浏览模式，承接"双击图片文件"这个系统级入口，让用户先看图、需要编辑时一步切入编辑器，体验对标 2345看图王这类被验证过的轻量看图软件，而不是重新发明。

### 12.2 核心设计原则（已讨论锁定，不要再重新论证）

1. **单 exe、内部双模式，不做两个独立程序**：浏览窗口和编辑窗口是同一进程内的两套界面，右键"编辑"是进程内窗口切换，不是 `Process.Start` 拉起新进程。参考 IrfanView / HoneyView / FastStone / XnView / 2345看图王本身的架构都是如此；Adobe Bridge→Photoshop 那种跨进程方案只适合本身启动就慢、用户对"重"有心理预期的专业软件，与 PicMark"小体积、秒开"的定位相反。
2. **不引入新的运行时依赖**：缩略图网格、虚拟化滚动、方向键切换全部用现有 WPF 能力实现（`VirtualizingStackPanel` + `BitmapImage.DecodePixelWidth` 做异步缩略图解码），不需要 OpenCV/AI 模型这类重依赖（这类方案在讨论"水印自动识别"时已被评估并否决，见 [[picmark-image-browser-plan]] 关联讨论）。
3. **接管系统默认双击打开方式**：用户明确选择"彻底接管"而非"只加右键菜单选项"。Win10/11 上应用无法静默抢占默认关联，只能引导用户跳转系统设置页（`ms-settings:defaultapps`）手动确认一次；Win7 无此限制，安装时可直接写注册表生效。这个平台差异需要在安装包/首次运行流程里分别处理，不能假设两个系统行为一致。
4. **不破坏现有编辑器的无损/不覆盖原图原则**：浏览模式本身是只读浏览，不涉及像素处理，不需要重新论证 [[picmark-architecture]] 里的无损约束，但浏览器里如果做"旋转"之类的操作，要确保走的是编辑器已有的同一套无损处理路径，不要在浏览模式里另起一套图像处理逻辑。

### 12.4 关键用户流程

```
系统双击图片文件
  → PicMark 以浏览模式启动，自动展示该文件所在文件夹的缩略图网格，并高亮选中被双击的那张
浏览
  → 点击/方向键切换缩略图，双击进入大图查看
（可选）编辑
  → 右键选择"编辑" → 同进程切换到编辑器界面，标注/裁切照旧
保存并返回
  → 编辑器保存后关闭 / 点击"返回浏览" → 切回浏览窗口，停留在原文件夹
```

### 12.5 技术方案要点

- 缩略图解码使用 `BitmapImage` + `DecodePixelWidth`（避免按原始分辨率全量解码再缩放，大幅降低内存和耗时），并放入后台线程异步加载，网格用虚拟化容器只渲染可视区域内的项。
- 浏览窗口与编辑器窗口是同一 `Application` 进程内的两个 `Window`，通过启动参数（命令行图片路径）或内部导航方法决定初始展示哪一个；不新增第二个项目/exe。
- 文件关联注册沿用现有 Inno Setup 安装包里"用简标打开"右键菜单的同一套机制（见 PRD 进度记录第 4 条），扩展为同时设置默认关联；Win10/11 静默接管受限的部分，改为安装完成后弹出引导提示，跳转系统默认应用设置页。
